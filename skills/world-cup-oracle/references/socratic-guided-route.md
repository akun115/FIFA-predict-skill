# Socratic Guided Route for World Cup Prediction

这是一条可选使用路线，用于用户希望先通过追问澄清判断，再查看模型预测的场景。它改进的是人机交互与赛前判断校准，不改变 `predict_match` 的概率来源。

## 触发条件

启用这条路线，当用户表达以下意图之一：

- “先别直接给结论，帮我想清楚这场怎么看。”
- “用苏格拉底式追问带我分析。”
- “我想自己先判断，再看模型。”
- “帮我找出我对这场比赛判断里的盲点。”
- “模型可能会在哪些地方错？”

如果用户只要快速预测，走标准 `world-cup-oracle` 流程即可。

## 设计原则

1. **先让用户形成判断，再展示模型概率。** 不要一开始就用模型结论压住用户思考。
2. **问题少而准。** 每轮最多 1-2 个问题，优先追问定义、证据、假设、反例和影响。
3. **事实必须核验。** 用户说出的伤停、阵容、状态、赔率或新闻都只是待核验输入；联网核验后才能写成事实。
4. **用户判断不进模型。** `[USER_PRIOR]` 只用于对照与校准，不得调 `predict_match` 概率。
5. **不制造虚假冲突。** 如果用户判断与证据或模型一致，说明一致点即可；不要为了“追问效果”硬找分歧。
6. **不提供投注建议。** 可以讨论概率、风险、模型限制和市场对照，但不能建议下注金额或操作。

## 五段式路线

### 1. 判断框定

目标：把“怎么看这场”变成一个可校准的问题。

可问：

- 你真正想判断的是胜平负、比分区间、晋级概率，还是爆冷风险？
- 如果只能选一个最关键的不确定因素，你会选阵容、状态、战术 matchup、赛程体能，还是心理压力？
- 你希望这次分析回答“谁更可能赢”，还是“模型最可能忽略什么”？

产出：一句明确的分析问题。

### 2. 初始承诺

目标：在模型输出前记录用户自己的可检验判断。

可问：

- 在看模型前，你的初始判断是什么？请给一个方向和大概信心。
- 你认为最可能比分或比分区间是什么？为什么？
- 你最担心哪个前提被证伪后会改变判断？

内部记录：

```text
[USER_PRIOR]
claim: ...
confidence: low | medium | high
key_assumptions:
  - ...
```

### 3. 假设追问

目标：暴露用户判断背后的关键前提。

问题类型：

| 类型 | 用途 | 示例 |
|---|---|---|
| 澄清 | 把模糊词具体化 | 你说“状态好”，具体指近几场结果、xG、阵容完整度，还是对手强度？ |
| 假设 | 找隐藏前提 | 你默认这名球员会首发吗？如果不是，判断会变多少？ |
| 证据 | 区分事实和印象 | 这个判断来自哪类证据：近期比赛、官方伤停、教练发言，还是媒体预测？ |
| 反例 | 测试稳健性 | 有没有一种比赛脚本会让你的判断完全失效？ |
| 影响 | 连接到模型风险 | 这个因素会改变胜负方向，还是只改变比分尾部风险？ |

### 4. 证据校准

目标：联网核验后，用证据挑战或支持用户判断。

执行顺序：

1. 用 `search_prematch` 生成检索计划。
2. 联网核验阵容、伤停、赛程、场地、天气、教练声明和必要背景。
3. 把信息分为：
   - **已核验事实**：有可靠来源支持。
   - **不确定信息**：来源不足、口径冲突或仍待确认。
   - **用户假设**：用户判断中尚未被证实的前提。
4. 如果证据与 `[USER_PRIOR]` 冲突，用追问方式呈现：
   - “如果这个伤停消息属实，你会把信心从 high 调到哪里？”
   - “这个反例是否影响你的主判断，还是只影响比分分布？”

### 5. 模型对照与收束

目标：用 `predict_match` 给出正式概率，并把用户判断放在校准框架中解释。

执行：

1. 调用 `predict_match`。
2. 原样保留模型输出：`model_status`、`model_version`、`data_quality`、xG、胜平负概率、Top 5 比分。
3. 对照：
   - 用户初始判断与模型一致在哪里？
   - 分歧在哪里？
   - 分歧来自事实输入、战术解释、样本限制、还是用户风险偏好？
4. 输出“判断校准摘要”。

推荐输出块：

```markdown
### 判断校准摘要
- 你的初始判断：...
- 核验证据后的变化：...
- 与模型一致：...
- 与模型分歧：...
- 我会把这个分歧视为：模型限制 / 信息缺口 / 战术不确定 / 用户风险偏好
```

## 失败与退出规则

- 用户要求“直接给预测”时，立即切回标准预测流程。
- 用户连续两轮无法给出判断时，可以提供选项式问题，但不要替用户下结论。
- 如果联网证据不足，明确报告缺口，不要用追问填补事实。
- 如果用户把路线用于下注决策，重申只提供分析和不确定性说明，不提供投注操作建议。
- 如果追问超过 6 轮仍未收束，汇总已有 `[USER_PRIOR]` 与证据缺口，建议进入标准预测。
